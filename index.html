We tried this too. Twice. The first one was called Striker — it died. The second one is called Coder — it works. Let me start with Striker. The architecture was simple but real. You at-mention striker in Slack. A Vercel function receives the webhook, provisions a fresh Vercel Sandbox per task, and drops in OpenCode — the open-source coding-agent CLI — running Claude. The sandbox clones the three repos the agent needs: the frontend, the agent itself, and the backend. OpenCode sees a bunch of tools — GitHub App for code, Linear for tickets, Slack for replies, plus Datadog, PostHog, LaunchDarkly, Figma. Every one of those tokens lives as an env var on the Vercel project. The agent does its work and opens a PR. The PR identity is our GitHub App — striker-bot. A human reviews and merges. Striker shipped fast. One Slack message turned into a sandboxed coding agent in thirty seconds, and a PR in five minutes. Zero local setup. Anyone could close a bug in their first week. Three weeks ago. April 19, 2026. Vercel discloses a security incident. Attackers got the environment variables on Vercel — API keys, tokens, database credentials. Plaintext. We weren't in the affected set. But our agent's tokens live there. The agent with write access to our repos. The agent that can reach into our infrastructure. And here's the kicker — read the bulletin. The attacker got in through a third-party AI tool that one Vercel employee was using. That tool compromised the employee's Google Workspace, which let them pivot into Vercel's internal systems. Sit with that for a second. A talk about coding agents starts with: "the breach started in someone else's coding agent." We dodged it. The next one might land. That was the day Striker had to die. After April — and after we admitted the sandbox couldn't run our stack — we replaced Striker with this. Coder. Built on coder.com — the open-source product. We host it ourselves, inside our Kubernetes cluster, inside our Tailscale mesh. You open coder.com in your browser, click "open workspace." A privileged Kubernetes pod boots — Docker-in-Docker, our full monorepo cloned, our frontend cloned. Two tmux sessions come up. The backend is running. The frontend is running. Claude Code is sitting in a tmux waiting for you. You type into Claude Code in your terminal — same as you would locally. When Claude Code opens a PR, it doesn't have a bot token. We wrap the gh CLI so it pulls your GitHub OAuth on every call. The PR opens in your name. Everything about the dev loop is better. The full stack runs end-to-end. Zero credentials baked in. You author the PR. The agent is your tool. [BRIDGE: Three things changed between Striker and Coder. Let me walk through them.]

Point one. If the agent can't run your stack, every PR is fiction. Striker on the left. We cloned our monorepo into a Vercel sandbox. But the config literally said install: skip on the backend. The agent was writing code against a backend that never started. Every PR was unverified fiction until a human pulled it locally. Coder on the right. Every workspace is a privileged Kubernetes pod running Docker-in-Docker, with our monorepo and frontend pre-cloned. When the workspace boots, two tmux sessions come up — one running the backend, one running the frontend. The agent ships nothing it didn't run end-to-end. TWO · HARD GUARDRAILS

The guardrail is the absence, not the instruction.

Prompt the agent never to open a PR.
Don't give the token PR-write scope.
Tell it not to touch the production DB.
Don't route prod DB into the container.
Hope the system prompt holds.
Make the wrong action physically impossible.
STRIKER
CODER
Env vars on Vercel. MCPs through vendors.
Zero credentials baked in. Tailscale + SSO for what you need.
Broad scopes, hope-driven.
Every cred faced one question: "jailbroken tomorrow — what does it do?" Answer: exfil customer data or delete prod? It didn't go in.
Point two. The guardrail is the absence, not the instruction. Three pairs. You can prompt the agent never to open a PR. Or you can withhold PR-write scope from the token. One is hope. One is a guarantee. You can tell the agent not to touch production. Or keep production out of the container. You can hope the system prompt holds. Or you can make the wrong action physically impossible — at the IAM layer, at the network layer. Remember the April bulletin? This is exactly why the credential model looks the way it does. Striker had env vars on Vercel and MCPs hosted by vendors. If a Vercel CVE had landed on us, every one of those tokens would be in the blast radius. Coder starts with zero credentials baked in. If you need DB access, you authenticate interactively — Tailscale plus SSO, like a human would. And we ran every credential through one question: if Coder gets jailbroken tomorrow, what does it do with this? For most things, fine. For a few, the answer was "it exfils customer data" or "it deletes prod." Those didn't go in. Not because we told it not to. Because they aren't there. AI-GENERATED CODE · STACK OVERFLOW + CODERABBIT · 2025

Last December, Amazon gave their AI coding agent Kiro a task: fix a small bug in AWS Cost Explorer. Kiro decided the cleanest path was to delete the production environment and rebuild it. Executed faster than a human could read a confirm prompt. Thirteen-hour outage. Three months later — March 2026 — two more outages, six hours each. Amazon dot com goes down. 6.3 million lost orders. The internal memo that leaked talked about a "trend of incidents," a "high blast radius," "GenAI-assisted changes." That's the alternative to building this carefully. That's what's been happening at one of the most sophisticated engineering orgs on the planet. And it's not just Amazon. CodeRabbit and Stack Overflow studied this last year. AI-generated code introduces 1.7x more bugs than human code. Security flaws at one and a half to two times the rate. I/O performance issues at eight times. The internet, measurably, broke more in 2025 than any prior year. [BRIDGE: Which is why we wrote this down as a single sentence.]
